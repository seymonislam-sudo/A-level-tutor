Programming Project Guide

The Non-Examined Assessment is worth 20% of your A Level. This guide walks through every section of the project — what AQA want, how marks are awarded, and how to avoid the most common pitfalls.

75 marks total
20% of A Level
AQA 7517/C

Overview & Mark Scheme

The NEA is an independent programming project completed during Year 13. You choose a real problem, develop software to solve it, and produce a written report documenting every stage. AQA assess both the report and the working program.

```
Analysis12 marks
Design12 marks
Implementation30 marks
Testing12 marks
Evaluation9 marks
SectionMarksWhat's assessed
Analysis12Problem identification, research, requirements, success criteria
Design12Data structures, algorithms, UI design, testing strategy
Implementation30Quality of code, complexity, use of appropriate techniques
Testing12Evidence of systematic testing against the design
Evaluation9Honest assessment of success, limitations, future improvements
The most important thing to understand Implementation carries 40% of the marks — but you cannot score highly in other sections without a working, complex program. A mediocre program with good documentation will score less than a sophisticated program well-documented. Both matter. Start coding early.
```

Choosing a Project

Your project choice largely determines your ceiling. A project that is too simple cannot earn high marks in implementation no matter how well documented. Aim for something genuinely complex — but manageable in your timeframe.

```

What makes a good NEA project?

✓
Has a real user with identifiable needs — not just "me" testing my own program
✓
Requires non-trivial algorithms — sorting, searching, pathfinding, parsing, simulation
✓
Uses appropriate data structures — linked lists, trees, hash tables, 2D arrays
✓
Has a clear problem that existing software doesn't already solve for your user
✓
Is achievable in the time available — don't aim for a commercial-grade product
✓
Allows you to demonstrate file I/O, data persistence, and a user interface

Project idea categories

Games

Puzzle games, strategy games, simulations. Good for algorithms (AI, pathfinding, collision detection) but avoid simple clones with no complexity.

Data management

Stock systems, library management, patient records. Good for file I/O, searching, sorting and data structures.

Tools & utilities

Revision quiz app, timetable generator, budget tracker. Clear user need; measurable success criteria.

Simulations

Physics simulations, ecosystem models, network simulators. Strong algorithmic content; impressive demonstrations.

Language tools

Simple interpreter, expression parser, code formatter. Demonstrates formal language theory from Topic 06.

Data analysis

Sports statistics analyser, weather data visualiser. Good for file handling, aggregation and graphical output.

Projects to avoid Website-only projects (limited algorithmic complexity); simple CRUD apps without any interesting logic; projects using low-code tools or frameworks that do the hard work for you; projects so ambitious they won't be finished. Also avoid projects identical to what your teacher has shown you — AQA moderators see these every year.
```

Analysis

```
Section A — Analysis
12 marks

AQA want to see that you understand the problem deeply before writing a single line of code. This section should convince the examiner you know exactly what you're building and why.

What to include

Problem identification: Describe the problem clearly. Who is the end user? What do they currently do? Why is a new solution needed? Be specific — avoid vague statements like "my user wants a program to manage things."

Stakeholder research: Interview or survey your user. Show evidence — include questions asked and responses received. This makes your requirements grounded rather than invented.

Research into existing solutions: Look at similar software. What do they do well? What don't they do? How is your solution different? Screenshots and analysis of real products score well here.

Requirements: Produce a numbered list of specific, measurable requirements. Use "The system shall..." phrasing. Split into functional (what it does) and non-functional (performance, usability, security).

Success criteria: For each requirement, state how you will test whether it has been met. These criteria will be referenced back to in your evaluation.

The success criteria link everything together Your success criteria from analysis should appear in your testing section as the basis for your test plan, and in your evaluation as the measure of whether you succeeded. Examiners look for this thread running through the whole project.
``` Example success criteria: SC1 The system shall allow users to add a new book record with title, author, ISBN and genre. Test: Add a book with valid data → verify record appears in search results. SC2 The system shall reject ISBN numbers that do not conform to the 13-digit ISBN-13 format. Test: Enter “12345” as ISBN → verify appropriate error message displayed. SC3 The system shall perform a search across 1000 records in under 1 second. Test: Load test dataset, time 10 searches, verify average < 1000ms.

Design

```
Section B — Design
12 marks

Design is your blueprint — produced before implementation. Examiners check that your design actually reflects what you built. Don't write design retrospectively after coding.

What to include

Decomposition: Break the problem into modules/functions. A structure chart or hierarchy diagram showing how components relate is ideal.

Data structures: Specify what data structures you'll use and why. If you're using a dictionary or list, explain why that's more appropriate than an array. If using a database, design your schema here with ER diagrams.

Algorithms: Write pseudocode or flowcharts for your key algorithms — especially any sorting, searching or complex logic. Don't pseudocode every function; focus on the non-trivial ones.

UI design: Sketches or wireframes of every screen/window. Annotate them to explain design choices. Show navigation between screens.

File/database design: If persisting data, design the file format or table structures. For a database: table names, field names, data types, PKs and FKs.

Testing strategy: Describe your overall approach — unit testing, integration testing, user acceptance testing. List test categories (normal, boundary, erroneous).

Design must come before implementation AQA penalise designs that are clearly written after the code — they look too perfect and exactly match what was built. A good design has some differences from the final product, with explanations of why changes were made.
```

Implementation

```
Section C — Implementation
30 marks

The largest section. Your actual code and a written commentary explaining key sections. The program must be functional — a non-working submission is severely penalised.

Mark bands — implementation

BandMarksDescription
4 (top)25–30Complex, well-structured solution. Sophisticated algorithms. Appropriate data structures. Clear, well-commented code. Evidence of iterative development.
319–24Reasonably complex solution. Some sophisticated elements. Mostly well-structured. Some commentary.
213–18Working but simple solution. Limited use of data structures. Basic algorithms only.
17–12Partially working solution. Significant gaps in functionality.
00–6Little working code. Program does not meet requirements.

What earns top marks

✓
Use of multiple appropriate data structures (not just lists for everything)
✓
Non-trivial algorithms — your own implementation, not library calls
✓
Object-oriented programming with appropriate use of inheritance and encapsulation
✓
File I/O or database persistence — data survives between sessions
✓
Robust error handling — program doesn't crash on invalid input
✓
Clear, consistent code style — meaningful names, appropriate comments
✓
Evidence of iterative development in your commentary (what you changed and why)

The written commentary

For key sections of your code, include a written explanation in your report. Don't narrate every line — explain the interesting decisions: why did you choose this algorithm? Why this data structure? What was the hardest part to implement and how did you solve it?

Show your code working Include annotated screenshots of your program running. Show the UI, show data being saved and loaded, show your key features in action. Examiners cannot run your program during moderation — screenshots are their window into what it does.
```

Testing & Evaluation

```
Section D — Testing
12 marks

Systematic evidence that your program works correctly. A test table with expected and actual results, plus screenshots. Tests must cover all three categories.

Test categories (all three required)

CategoryDescriptionExample
Normal / validTypical inputs the program should handle correctlyAdd a book with valid ISBN "9781234567897"
BoundaryValues at the edges of acceptable rangesUsername of exactly 3 characters (minimum allowed); username of exactly 20 (maximum)
Erroneous / invalidInputs the program should reject with appropriate errorISBN containing letters; negative price; empty required field
``` Test table format: Test ID | Description | Input data | Expected result | Actual result | Pass/Fail | Screenshot T01 | Add valid book | Title=“Harry P.. | Record appears in list | As expected | Pass | Fig. 12 T02 | Boundary: min ISBN | ISBN=“1234567890128” | Accepted | As expected | Pass | Fig. 13 T03 | Invalid ISBN (alpha) | ISBN=“ABCDEFGHIJK” | Error: “Invalid ISBN” | As expected | Pass | Fig. 14
```
Section E — Evaluation
9 marks

An honest assessment of your project against your original success criteria. Reference each criterion by number. Acknowledge what doesn't work — pretending everything is perfect scores badly.

What to include in evaluation

Success criteria review: Go through each success criterion. Did you meet it? Provide evidence from your test results. If partially met, explain what works and what doesn't.

User feedback: Return to your stakeholder. Show them the finished product. Record their feedback. Did it solve their problem? What would they change?

Limitations: Be specific and honest. "The search algorithm is O(n) — for large datasets a binary search or hash table would be more efficient." This shows technical understanding.

Future improvements: Prioritised list of what you would add given more time. Be specific — not "I would add more features" but "I would implement a BST for O(log n) search performance."

Reflection on design decisions: What would you do differently? What worked better than expected?

```

Suggested Timeline

The NEA is typically completed across the autumn and spring terms of Year 13. Most schools allocate September to February — roughly 20 weeks.

```
WeeksPhaseTarget deliverable
1–2Project selectionApproved project idea; identified stakeholder
3–5AnalysisCompleted analysis section with requirements and success criteria
6–8DesignCompleted design section with UI mockups, data structures and algorithms
9–16ImplementationWorking program with all core features; iterative development log
17–18TestingCompleted test table with screenshots; all three test categories
19–20Evaluation & polishCompleted evaluation; report proofread; all sections checked against mark scheme
Don't leave implementation until week 9 Start prototyping early — even during analysis. Understanding the technical challenges shapes your requirements. Many students discover their original plan is too ambitious (or too simple) only when they start coding. Fail fast, adjust early.
```

Examiner Tips

```

Maximising your marks

✓
Reference the mark scheme throughout. AQA publish the assessment objectives — know them and explicitly address each one.
✓
Use technical vocabulary. Say "binary search tree" not "a type of sorted list." Say "O(n log n) time complexity" not "fast sorting."
✓
Link sections together. Your evaluation should reference your success criteria (from analysis) and your test results (from testing). Examiners reward coherence.
✓
Include evidence of iteration. Show that you built, tested, improved and rebuilt. A version history or development log demonstrates the process.
✓
Get real user feedback. Not from a parent or friend — from someone who would actually use the software. Record their exact words.
✓
Comment your code meaningfully. Not # add 1 to count but # increment pointer to next node in linked list. Explain the why, not the what.
✓
Back up constantly. Use Git. Losing your work near the deadline has happened to students before. Don't be one of them.

Common mistakes to avoid

✗
Writing the design section after the code is written — examiners can tell
✗
Including all your code in the report — only include and annotate key excerpts
✗
Test tables with no "actual result" or "pass/fail" column — or no screenshots
✗
Evaluation that claims everything is perfect — no real project has no limitations
✗
Choosing a project so simple it can't score above band 2 in implementation
✗
Submitting a program that crashes or has major bugs — test thoroughly before submission
✗
Using AI tools to write your code — AQA and your school take academic integrity seriously
```

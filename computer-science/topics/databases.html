Relational Databases

A relational database organises data into tables (relations). Each table has rows (records/tuples) and columns (attributes/fields). Tables are linked through keys.

```

Key Terminology

TermDefinition
Table / RelationA named set of rows and columns representing an entity
Row / Tuple / RecordA single entry in a table — one instance of the entity
Column / Attribute / FieldA category of data stored in a table
Primary Key (PK)A field (or combination) that uniquely identifies each row — must be unique and not null
Foreign Key (FK)A field in one table that references the primary key of another — creates a relationship
Composite KeyA primary key made of two or more fields
EntityA real-world object or concept represented by a table (e.g. Student, Order, Product)

Referential Integrity A foreign key value must either match an existing primary key in the referenced table, or be NULL. The database enforces this — you cannot add an order for a customer that doesn't exist, and you cannot delete a customer who has orders.

Normalisation

Normalisation is the process of structuring a database to reduce data redundancy and improve integrity. AQA requires knowledge up to Third Normal Form (3NF).

```

Problems Normalisation Solves

AnomalyDescription
Update anomalyChanging one piece of data requires updating it in multiple places — risk of inconsistency
Insertion anomalyCannot add data about one entity without having data about another (e.g. can't add a course without enrolling a student)
Deletion anomalyDeleting one record accidentally removes other needed data

First Normal Form (1NF)

A table is in 1NF if: each cell contains a single atomic value (no lists), all values in a column are the same type, each row is unique, and there are no repeating groups of columns.

Violates 1NF OrderID | Products: "Pen, Ruler, Book" — the Products column contains multiple values. Fix: one row per product.

Second Normal Form (2NF)

A table is in 2NF if it is in 1NF AND every non-key attribute is fully functionally dependent on the whole primary key. Only applies when there is a composite key — partial dependencies must be removed.

Worked Example — 1NF → 2NF
Table: OrderLine(OrderID, ProductID, ProductName, Quantity). The PK is (OrderID, ProductID). ProductName depends only on ProductID — a partial dependency. Fix it.
1
Identify the partial dependency: ProductName → ProductID only (not the full composite key)
2
Split into two tables:
OrderLine(OrderID, ProductID, Quantity) — PK is composite
Product(ProductID, ProductName) — PK is ProductID

Third Normal Form (3NF)

A table is in 3NF if it is in 2NF AND there are no transitive dependencies — no non-key attribute depends on another non-key attribute.

Worked Example — 2NF → 3NF
Table: Student(StudentID, CourseID, CourseName, LecturerID, LecturerName). CourseName and LecturerName depend on CourseID, not StudentID. LecturerName depends on LecturerID — a transitive dependency.
1
Identify transitive dependency: LecturerName depends on LecturerID (not StudentID)
2
Split into:
Enrolment(StudentID, CourseID)
Course(CourseID, CourseName, LecturerID)
Lecturer(LecturerID, LecturerName)
Summary of Normal Forms 1NF: atomic values, no repeating groups.
2NF: 1NF + no partial dependencies (non-key attributes depend on whole PK).
3NF: 2NF + no transitive dependencies (non-key attributes only depend on the PK, not each other).
```

SQL Queries

SQL (Structured Query Language) is the standard language for querying and manipulating relational databases. AQA requires SELECT, INSERT, UPDATE, DELETE, and table creation.

```

SELECT

``` – Basic SELECT SELECT name, age FROM Students; – SELECT all columns SELECT * FROM Students; – WHERE clause (filtering) SELECT name, grade FROM Students WHERE grade = ‘A’; – Multiple conditions SELECT name FROM Students WHERE age >= 16 AND grade IN (‘A’, ‘A*’); – LIKE for pattern matching SELECT name FROM Students WHERE name LIKE ‘A%’; – Names starting with A – ORDER BY SELECT name, grade FROM Students ORDER BY grade DESC; – DISTINCT removes duplicates SELECT DISTINCT subject FROM Enrolment;
```

INSERT, UPDATE, DELETE

``` – INSERT a new row INSERT INTO Students (StudentID, name, age, grade) VALUES (‘S003’, ‘Emma’, 17, ‘B’); – UPDATE existing rows UPDATE Students SET grade = ‘A’ WHERE StudentID = ‘S003’; – DELETE rows DELETE FROM Students WHERE StudentID = ‘S003’;
```
Always use WHERE with UPDATE and DELETE Without a WHERE clause, UPDATE changes every row and DELETE removes every row in the table. This is an extremely common and costly mistake. Always double-check your WHERE conditions.

CREATE TABLE

``` CREATE TABLE Students ( StudentID VARCHAR(10) PRIMARY KEY, name VARCHAR(100) NOT NULL, age INTEGER, grade CHAR(2), CourseID VARCHAR(10), FOREIGN KEY (CourseID) REFERENCES Courses(CourseID) );

```

JOINs

A JOIN combines rows from two tables based on a related column. AQA focuses on INNER JOIN and LEFT JOIN.

JOIN typeReturns
INNER JOINOnly rows where the join condition matches in BOTH tables
LEFT JOINAll rows from left table + matching rows from right (NULL if no match)
RIGHT JOINAll rows from right table + matching rows from left
``` – INNER JOIN: students with their course names SELECT Students.name, Courses.CourseName FROM Students INNER JOIN Courses ON Students.CourseID = Courses.CourseID; – LEFT JOIN: all students, even those not enrolled SELECT Students.name, Courses.CourseName FROM Students LEFT JOIN Courses ON Students.CourseID = Courses.CourseID; – Three-table JOIN SELECT s.name, c.CourseName, e.grade FROM Students s INNER JOIN Enrolment e ON s.StudentID = e.StudentID INNER JOIN Courses c ON e.CourseID = c.CourseID WHERE e.grade = ‘A*’ ORDER BY s.name;
```

Aggregate Functions & GROUP BY

``` – Aggregate functions SELECT COUNT(*) FROM Students; – count all rows SELECT AVG(age) FROM Students; – average age SELECT MAX(age), MIN(age) FROM Students; – max and min SELECT SUM(marks) FROM Results; – total marks – GROUP BY: count students per grade SELECT grade, COUNT(*) AS num_students FROM Students GROUP BY grade ORDER BY grade; – HAVING filters groups (like WHERE but for aggregates) SELECT CourseID, AVG(grade) AS avg_grade FROM Enrolment GROUP BY CourseID HAVING AVG(grade) > 70;
```
Worked Example
Using the Student, Enrolment, and Course tables, write a query to find the name and average grade of every student who is enrolled in more than one course, ordered by average grade descending.
``` SELECT s.name, AVG(e.grade) AS avg_grade, COUNT(*) AS num_courses FROM Students s INNER JOIN Enrolment e ON s.StudentID = e.StudentID GROUP BY s.StudentID, s.name HAVING COUNT(*) > 1 ORDER BY avg_grade DESC;

Key points: GROUP BY includes all non-aggregate SELECT columns; HAVING filters after grouping (WHERE filters before); AVG and COUNT are applied per group.

Transactions & ACID

A transaction is a sequence of database operations treated as a single unit. Either all operations succeed (commit) or none take effect (rollback). This prevents partial updates corrupting the database.

```
``` – Bank transfer: debit one account, credit another BEGIN TRANSACTION; UPDATE Accounts SET balance = balance - 100 WHERE AccountID = ‘A001’; UPDATE Accounts SET balance = balance + 100 WHERE AccountID = ‘A002’; COMMIT; – Both succeed: save permanently – ROLLBACK would undo both if an error occurred
```

ACID Properties

PropertyMeaningBank transfer example
AtomicityAll operations in a transaction succeed, or none do — no partial completionBoth debit and credit happen, or neither does
ConsistencyThe database moves from one valid state to another — all rules and constraints are maintainedTotal money in system stays the same
IsolationConcurrent transactions don't interfere — each appears to run aloneTwo transfers don't read each other's intermediate states
DurabilityOnce committed, changes survive crashes — persisted to diskCompleted transfer is not lost if server crashes
Record Locking To maintain Isolation, databases use record locking — a lock is placed on records being modified so other transactions must wait. This prevents two users updating the same record simultaneously and causing data corruption. A deadlock occurs when two transactions each hold a lock the other needs — detected and resolved by rolling back one transaction.
```

Abstraction & Decomposition

Computational thinking is the process of formulating problems so they can be solved by a computer. It has four key components:

```
ComponentDefinitionExample
AbstractionRemoving unnecessary detail, keeping only what mattersA map shows roads but not every blade of grass
DecompositionBreaking a complex problem into smaller, manageable sub-problemsA school system split into: students, lessons, timetable, staff
Pattern recognitionIdentifying similarities and common features in problemsSorting algorithms all compare and swap elements
Algorithmic thinkingDesigning step-by-step solutions that can be followed preciselyA recipe, navigation directions

Levels of Abstraction

Abstraction exists at multiple levels in computing. Understanding which details to hide at each level is a key skill:

Examples of Abstraction Levels High-level language → Assembly language → Machine code → Logic gates → Transistors. Each level hides the complexity below it. A Python programmer doesn't need to know about transistors; a hardware engineer doesn't need to know Python.
Worked Example
A developer is building a sat-nav app. Identify two abstractions they would use and explain what detail each one removes.
1
Map abstraction — roads shown as lines, removing physical width, surface texture, buildings alongside the road, etc.
2
Route abstraction — directions given as turns only, hiding the complexity of GPS coordinates, triangulation, signal processing.
```

Algorithms & Flowcharts

An algorithm is a finite, precise, step-by-step set of instructions to solve a problem. Algorithms can be expressed as pseudocode, flowcharts, or program code.

```

Flowchart Symbols

SymbolShapeMeaning
TerminalRounded rectangle (oval)Start or End
ProcessRectangleAn instruction or calculation
DecisionDiamondA yes/no question (branch point)
Input/OutputParallelogramData input or output
SubroutineRectangle with double vertical linesCall to a subroutine

Properties of a Good Algorithm

Three Essential Properties 1. Correctness — produces the right output for all valid inputs.
2. Efficiency — uses minimal time and memory resources.
3. Clarity — easy to understand and maintain.
Worked Example — Pseudocode
Write pseudocode for an algorithm that finds the largest number in a list of n numbers.
``` largest ← list[0] FOR i ← 1 TO LENGTH(list) - 1 IF list[i] > largest THEN largest ← list[i] ENDIF ENDFOR OUTPUT largest

We initialise largest to the first element, then iterate through the rest, updating largest whenever we find a bigger value.

Finite State Machines

A Finite State Machine (FSM) is a model of computation consisting of a finite number of states, transitions between those states, and inputs that trigger transitions.

```

Key Terminology

TermDefinition
StateA condition the system can be in at a given moment
TransitionA change from one state to another, triggered by an input
Start stateThe state the machine begins in (shown with an arrow)
Accept stateA state that means the input has been accepted (shown with double circle)
AlphabetThe set of valid input symbols

State Transition Diagrams

FSMs are drawn as state transition diagrams. States are circles, transitions are arrows labelled with the input that triggers them.

FSM with Output (Mealy Machine) Some FSMs produce output on each transition — called a Mealy machine. Others produce output based on state — called a Moore machine. AQA may ask you to complete or interpret either type.

Turing Machines

A Turing machine is a theoretical model of computation proposed by Alan Turing in 1936. It provides a formal definition of what it means for a problem to be computable.

```

Components of a Turing Machine

ComponentDescription
Infinite tapeDivided into cells, each holding one symbol. Extends infinitely in both directions.
Read/write headReads the symbol on the current cell, can write a new symbol, moves left or right
State registerHolds the current state of the machine
Transition functionA table of rules: given (current state, symbol read) → (write symbol, move direction, new state)
Halt stateA special state that stops the machine — either accept or reject
The Universal Turing Machine A Universal Turing Machine (UTM) can simulate any other Turing machine. This is the theoretical basis for the modern stored-program computer — any computation can be encoded as data and processed by a general-purpose machine.
Worked Example — Transition Table
A Turing machine is designed to add 1 to a binary number on the tape. The head starts at the rightmost digit. Complete the transition table.
StateReadWriteMoveNew State
q0 (adding)10Leftq0 (carry)
q0 (adding)01Leftq1 (done)
q0 (adding)B (blank)1Leftq1 (done)
q1 (done)anyunchanged—HALT

The machine moves left, flipping 1s to 0s (carrying) until it finds a 0 or blank, which it writes as 1 and halts.

```

Complexity & Decidability

```

The Halting Problem

Alan Turing proved in 1936 that it is impossible to write a general algorithm that determines, for any arbitrary program and input, whether that program will eventually halt or run forever. This is known as the Halting Problem.

Why it matters The Halting Problem shows that there exist problems that are undecidable — no algorithm can ever solve them for all inputs. This is a fundamental limit of computation itself, not just of current technology.

Decidable vs Undecidable Problems

TypeDefinitionExample
DecidableAn algorithm exists that always gives a yes/no answer in finite timeIs a number prime? Is a string a palindrome?
UndecidableNo algorithm can solve it for all possible inputsThe Halting Problem, some tiling problems

Tractable vs Intractable Problems

Even among decidable problems, some are tractable (solvable in reasonable time) and some are intractable (theoretically solvable but impractical for large inputs).

TypeTime ComplexityExample
TractablePolynomial time — O(n), O(n²), O(n³)Sorting, searching, shortest path
IntractableExponential time — O(2ⁿ), O(n!)Travelling Salesman Problem (exact), factoring large numbers
Exam Trap Students often confuse undecidable (no algorithm exists) with intractable (algorithm exists but is too slow). Make sure you use the correct term. The Halting Problem is undecidable — not just intractable.
Worked Example
Explain why the Travelling Salesman Problem (finding the shortest route visiting n cities exactly once) is considered intractable.
1
The number of possible routes for n cities is (n-1)! — a factorial growth rate.
2
For 20 cities: 19! ≈ 1.2 × 10¹⁷ routes. Even checking one million routes per second would take millions of years.
3
An exact algorithm exists but is impractical for large inputs — so the problem is intractable, not undecidable. Heuristic approaches give good (not guaranteed optimal) solutions in reasonable time.
```
